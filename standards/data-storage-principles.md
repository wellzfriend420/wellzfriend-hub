# データ保存の基本原則

> 状態: approved / 管理責任: 情報管理責任者 / 承認責任: アプリ事業責任者 / 決定日: 2026-07-04 / 最終確認日: 2026-07-04

## 目的

設計情報、利用成果物、業務データ、開発途中の情報を混在させず、それぞれに一つの正本を定めます。新規アプリ、新規プロジェクト、既存資産の移行は本原則を共通ルールとして適用します。

基本原則は、**GitHubには「設計・開発・運営の知識」を保存し、Google Driveには「人が利用する成果物」を保存する**ことです。日々更新される業務データの正本は各会社の統合データベースとします。

## 保存場所と役割

### GitHub — 設計図書館

設計・開発・運営の知識を、変更履歴とともに管理する正本です。

保存対象:

- ソースコード
- README、README_AI
- アプリ事業運営マニュアル
- 設計書
- Hub
- 各種台帳（Registry）
- 開発標準
- 変更履歴
- テンプレートの設計情報

保存しないもの:

- Googleスプレッドシート、Googleドキュメント、Googleフォームの実体
- 日々更新される業務データ、顧客データ
- 契約書、提案書、納品物等の利用成果物の実体
- パスワード、APIキー、トークン、秘密鍵、DB接続情報

Google Driveや統合DB上の対象を参照するときは、秘密値を含まない管理IDまたは論理参照先だけを記録します。

### Google Drive — 業務書庫

人が実際に利用・共有・提出する成果物の正本です。

保存対象:

- 親テンプレートの実体
- Googleスプレッドシート
- Googleドキュメント
- Googleフォーム
- PDF
- 契約書、提案書、納品物
- ユーザーが実際に利用する成果物

テンプレートの設計方針・ID・Version・責任・論理参照先はGitHubのHubで管理し、利用する親テンプレートの実体はGoogle Driveで管理します。顧客情報を含む成果物には適切なアクセス制御、共有範囲、保持・削除方針を適用します。

### 各会社統合DB — データ保管庫

各会社で日々生成・更新される業務データの正本です。

保存対象:

- 顧客データ
- 売上、請求
- 工程、勤怠
- 在庫、仕入
- その他の業務データ

会社ごとに分離し、HubやGitHubへ実データを複製しません。Hubには管理ID、責任役割、利用アプリID、データ分類、保持・削除方針、バックアップ方針、秘密値を含まない論理参照先だけを保存します。

### ローカル（WellzFriend_OS）— 開発工房

GitHubへ反映する前の開発、テスト、検証を行う一時的な作業環境です。

保存対象:

- 開発中のコード・文書・設計情報
- テストと検証に必要な非本番データ
- GitHubへPushする前の作業内容

ローカルだけを確定情報の正本にしません。安定した設計・開発・運営知識は確認後にGitHubへ反映します。テストでは匿名化または合成データを使用し、顧客データや本番秘密情報を持ち込みません。

## 判断ルール

1. 新しい情報を作る前に「知識」「利用成果物」「業務データ」「一時作業」のどれかへ分類する。
2. 正本を一つ決め、他の保存場所には実体を複製せず管理IDまたは論理参照を置く。
3. Google形式の親テンプレートは、設計情報をGitHub、利用実体をGoogle Driveへ分離する。
4. 業務データを文書、Issue、ソースコード、テストログ、生成物へ転記しない。
5. 保存先にかかわらず、最小権限、保持・削除、バックアップ、管理責任、最終確認日を定める。
6. 分類に迷う場合は保存せず、情報管理責任者が正本と参照方法を決定する。

## 新規アプリ・プロジェクト開始時の確認

- [ ] GitHubリポジトリと管理責任を登録した
- [ ] Google Driveの成果物保存先と管理責任を決めた
- [ ] 業務データを扱う場合、会社統合DBの管理IDと方針を登録した
- [ ] ローカルに顧客データ・本番秘密情報を置かないテスト方法を決めた
- [ ] 各情報の正本と、他領域からの論理参照方法を文書化した
